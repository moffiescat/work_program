# 项目模块化架构设计

## 一、架构概述

本项目采用**模块化架构**，将五大子系统拆分为独立的模块，降低耦合度，便于团队并行开发。

```
com.example.work_program
├── common/                    # 公共模块（所有子系统共享）
│   ├── Result.java            # 统一响应格式
│   ├── annotation/            # 自定义注解
│   │   └── LoginRequired.java # 登录权限注解
│   └── exception/             # 公共异常处理（待扩展）
│
├── config/                    # 公共配置
│   ├── WebConfig.java         # Web配置（CORS、拦截器）
│   └── JwtConfig.java         # JWT配置（待扩展）
│
├── util/                      # 工具类
│   └── JwtUtil.java           # JWT工具类
│
├── modules/                   # 子系统模块
│   └── datacollection/         # 数据影像采集系统模块 ★（已完成）
│       # 其他模块待团队成员分别开发
│   ├── system/                 # 系统管理模块（待开发）
│   ├── elder/                 # 老人健康管理模块（待开发）
│   ├── risk/                   # 认知衰弱风险评估模块（待开发）
│   ├── intervention/           # 健康认知与干预管理模块（待开发）
│   └── analysis/               # 效果分析决策支持模块（待开发）
│
└── WorkProgramApplication.java
```

## 四、API接口规范

### 4.1 数据影像采集模块接口

| 接口路径 | 方法 | 说明 | 权限 |
|----------|------|------|------|
| `/api/data/collection/list` | GET | 查询采集记录 | 登录 |
| `/api/data/collection/{id}` | GET | 获取详情 | 登录 |
| `/api/data/collection/add` | POST | 添加记录 | doctor/nurse |
| `/api/data/collection/update` | PUT | 更新记录 | doctor/nurse |
| `/api/data/collection/delete/{id}` | DELETE | 删除记录 | doctor |
| `/api/data/collection/statistics` | GET | 统计数据 | 登录 |
| `/api/data/image/*` | | 影像报告CRUD | |
| `/api/data/smart/*` | | 智能评估CRUD | |
| `/api/data/questionnaire/*` | | 健康问询CRUD | |

### 4.2 响应格式

```json
{
    "code": 200,
    "message": "操作成功",
    "data": { ... }
}
```

## 五、团队协作规范

### 5.1 Git分支管理

```
master          # 主分支
├── dev         # 开发分支
│   ├── system  # 系统管理开发
│   ├── elder   # 健康档案开发
│   ├── datacollection  # 数据采集开发 ★
│   ├── risk    # 风险评估开发
│   ├── intervention  # 干预管理开发
│   └── analysis  # 效果分析开发
└── feature/*   # 功能分支
```

### 5.2 代码提交规范

```
[模块] 提交描述
示例：[datacollection] 添加影像报告上传功能
```

### 5.3 模块间调用

- **允许**：调用 `common/` 下的公共类
- **允许**：通过 `@Autowired` 注入其他模块的 `Service`
- **禁止**：直接访问其他模块的 `Controller`、`Mapper`
- **建议**：通过 API 接口进行模块间通信

### 5.4 数据库表规范

## 六、扩展指南

### 添加新模块

1. 在 `modules/` 下创建新模块目录
2. 按照 controller → service → mapper → entity 的层次结构创建代码
3. 在 `WorkProgramApplication.java` 所在包的子包中组织（Spring Boot自动扫描）
4. 更新本文档的模块职责表

### 模块独立打包（可选）

如需将模块打包为独立JAR，可在 `pom.xml` 中配置 Maven Module。
